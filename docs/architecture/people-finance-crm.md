# 人员、财务与 CRM

## 数据边界

`EmployeeProfile` 只保存工号、岗位、任职状态和与 `Member`/`Department` 的关系。法定姓名、证件、私人联系方式和紧急联系人放在独立的 `EmployeeIdentity`；基础薪资放在独立的 `EmployeeCompensation`。服务端按角色决定是否查询这两个关系，因此无权角色的 React Server Component 响应中不会出现敏感字段，而不只是隐藏表格列。

HR、Owner、Admin 可管理员工并查看身份资料；Finance 只能查看薪资与商务金额；Manager 只能审批自己所属部门的考勤并查看该部门的非敏感记录。所有写入口重新校验当前成员、组织和关联记录，页面可见性不作为授权边界。

`AttendanceException` 只记录异常与审批，不复制每日打卡。批准/拒绝采用带旧状态条件的更新，并在同一事务写 `AttendanceApprovalHistory` 与 `AuditEvent`。`MonthlyAttendance` 按员工、年、月唯一，由已批准异常重建；它只作为薪资参考，不产生自动扣薪。

`PayrollMonth` 按员工、年、月唯一保存基础薪资、绩效、佣金、补贴、扣罚和实发金额。金额使用 `Decimal(18,2)`，数据库约束实发金额必须等于各源字段之和减扣罚。`MajorError` 独立保留事实、审核状态与建议扣罚，是否进入当月扣罚由有权限人员明确决定。

## 外部主体与资金

`ExternalParty` 用角色数组统一客户、供应商、服务商和顾问。同一主体可兼有多个角色。迁移会原位重命名 Issue #4 的 `CommerceSupplier`，保留供应商 ID、商品外键和已有联系方式，再给旧记录补 `SUPPLIER` 角色，不复制外部主体。

该迁移依赖工作内容和直播电商迁移，目录时间戳必须晚于 `20260712170000_live_commerce`。本模块使用 `20260712180000_people_finance_crm`，确保空库按依赖顺序创建 `CommerceSupplier`/`CommerceProduct` 后再执行无损重命名。

`Contract` 保留合同编号、主体、负责人、日期、金额和受控状态历史。`PaymentItem` 表示一笔应收或应付计划，`PaymentRecord` 表示实际发生的分次收付款；余额由计划金额减付款记录汇总得到。付款登记使用可串行化事务和旧金额/旧状态条件更新，超额付款、重复终态操作与并发过期写入会失败。合同流转、审批和付款均写审计日志。

## 页面

- `/people`：员工基础档案；身份和薪资列按服务端权限查询。
- `/people/attendance`：异常申请、部门审批和月度汇总。
- `/people/payroll`：月度薪资与重大错误。
- `/business`：统一外部主体。
- `/business/contracts`：合同台账与状态流转。
- `/business/payments`：应收、应付、余额和分次收付款。

各路由均包含加载、空数据和错误恢复状态；移动端把复杂表格保留为横向滚动，把审批和付款表单改为单列。

## 仿真数据约定

统一种子脚本应只使用明显虚构的数据，并形成连贯关系：至少两个部门与不同角色成员；员工分别具备身份、薪资、已批准/待审批异常、月度考勤和月度薪资；至少一个重大错误；一个同时为客户和服务商的主体、一个由旧供应商模型迁移语义覆盖的供应商；有效与已完成合同；部分回款的应收和已付清的应付及其付款记录。不得出现原公司人员、品牌、合同编号或薪资。
