# 经营看板与固定自动化

## 聚合与权限

看板在请求时直接聚合业务源表，不维护第二套汇总表。日期筛选按组织 `timezone` 把本地自然日转换成 UTC 半开区间 `[from, to)`；内容按创建时间、直播按开播时间、项目按创建时间、人员薪资按记录时间、现金按截至区间末仍未结清或区间内有付款的款项统计。所有明细保留源记录 ID 并链接回业务页。

Owner、Admin 查看全部域；Manager 只查看自己部门负责人名下的内容、直播与项目；HR 只查看人员域；Finance 只查看现金域。普通成员不获得聚合数据。权限与组织隔离在服务端查询条件中执行。

金额始终由 Prisma `Decimal` 求和，退款率分母为零时返回零。页面使用原生指标卡和表格，未引入图表依赖。

## 自动化

`AutomationRule` 只保存五种代码内规则的启停、固定接收人和少量阈值配置，不提供通用工作流设计器：

- 直播前提醒负责人和跟播人员。
- 活跃合同到期提醒。
- 未处理负面评价提醒。
- 视频生产超过 48 小时 SLA 提醒。
- 项目逾期、应收未回款和应付待付款异常提醒 Owner。

一次 tick 由 `POST /api/internal/automations/tick` 触发，使用独立的 `AUTOMATION_SECRET` Bearer Token 保护。部署定时器由部署模块负责配置。每个规则每个 UTC 小时只有一个 `AutomationRun` 幂等键；业务通知另有“组织 + 规则 + 源事件 + 接收人”唯一键，因此重复 tick 或失败重试都不会重复通知。失败记录错误、完成时间和尝试次数，并可在 `/settings/automations` 查看和手动重试。

所有规则查询都带 `organizationId`，通知写入 Core 的 `Notification`，不引入 Redis 或独立任务后端。
