# Creator Ops 系统设计

## 1. 目标

把内容公司的公司管理表、短视频管理表与补充需求转化为一套可开源、可私有化部署的软件。系统面向内容公司，而不是复刻某一份多维表格。

首版必须全覆盖，但以一个模块化单体交付：各模块共享人员、权限、文件、评论、通知和审计能力，避免为每张原表复制一套 CRUD。

## 2. 明确边界

### 包含

1. 部门、成员和角色权限。
2. 项目与任务、AI 提效需求。
3. 短视频生产、发布、数据和工作量统计。
4. 视频号直播排期、基础数据和复盘。
5. 商品、直播商品表现、评论与客诉。
6. 公众号与图书排期。
7. 员工档案、考勤异常、月度考勤、薪资绩效和重大错误。
8. 客户、服务商、顾问、合同、应收与应付。
9. 部门看板与老板经营看板。
10. 直播提醒、合同到期提醒、负面评价预警和内容 SLA 预警。
11. Docker 私有化部署、虚构种子数据、备份说明和飞书迁移入口。

### 不包含

- 多租户 SaaS。
- 通用低代码表格、公式语言或工作流编辑器。
- 订单与退货逐笔明细。
- 自动发薪、自动考勤扣款或银行支付。
- 对真实飞书数据的公开导出。

## 3. 技术架构

- Web：Next.js App Router、TypeScript、React Server Components。
- 数据库：PostgreSQL；数据库是业务状态的唯一事实来源。
- 数据访问：Prisma，迁移文件随仓库提交。
- 身份认证：成熟认证库提供账号、密码散列和会话；应用层维护角色与业务权限。
- UI：Tailwind CSS；优先使用原生表单、CSS Grid 和少量可访问组件。
- 图表：只为经营看板引入一个图表库。
- 后台任务：首先使用应用内定时执行器与 PostgreSQL 任务表；只有实际吞吐证明不足时才增加独立队列。
- 文件：本地卷为默认实现，S3 兼容存储作为可选配置；数据库只存元数据。
- 部署：一个 Web 容器、一个 PostgreSQL 容器，不要求 Redis。

## 4. 模块边界

### Core

组织设置、部门、成员、角色、会话、文件、评论、通知、审计日志和系统配置。其他模块只引用成员与文件 ID，不复制用户资料。

### Work

项目、任务和 AI 提效需求。项目支持负责人、参与人、时间范围、状态、优先级与关联客户/服务商；任务支持父子关系但 UI 首版只展示一层。

### Content

统一内容主表承载标题、IP、内容类型、负责人、计划时间、发布状态和渠道。视频详情承载拍摄、剪辑、审核、优先级、SLA 与工作量积分；文章/图书详情承载选题、撰稿、审稿和排期。发布记录按渠道保存发布时间、账号与指标快照。

视频生产状态采用受控状态机：`IDEA -> READY_TO_SHOOT -> ROUGH_CUT -> FINE_CUT -> REVIEW -> SCHEDULED -> PUBLISHED`，另有 `RETAKE`、`REJECTED`、`ARCHIVED`。状态变更写入历史表，统计不依赖覆盖后的当前值。

### Live Commerce

直播场次、商品、直播商品表现、评价/客诉。直播场次可关联预热视频；商品表现按“场次 × 商品”记录曝光、点击、订单、成交、退款和佣金汇总。

### People & Finance

员工档案、考勤异常、月度考勤、薪资绩效、重大错误。薪资与身份资料属于敏感域，只有 owner/admin/HR 可访问；普通经理只能访问明确授权的部门汇总。

### CRM

客户、服务商/顾问、合同、应收与应付。项目、商品和直播通过关系表关联外部主体，不使用多态字符串外键。

### Automation & Analytics

自动化不是任意节点编排器，而是代码内注册的规则模板。数据库保存规则是否启用、执行时间和接收人；执行记录可审计、可重试。经营看板使用数据库聚合查询，不保存人工重复填写的汇总表。

## 5. 权限

系统角色：

- `OWNER`：全部权限及系统设置。
- `ADMIN`：除所有权转移外的管理权限。
- `HR`：员工敏感资料、考勤和薪资。
- `FINANCE`：合同、应收应付和薪资金额，不访问身份证等身份资料。
- `MANAGER`：被授权部门的业务与聚合数据。
- `MEMBER`：参与或负责的记录。
- `VIEWER`：被授权模块只读。

权限在服务端查询与变更入口执行；隐藏前端按钮不能代替鉴权。敏感字段不进入无权限用户的响应。

## 6. 关键数据规则

- 金额使用数据库 Decimal 和明确币种，禁止浮点数计算。
- 日期时间以 UTC 存储，组织配置保存展示时区。
- 软删除只用于需要审计恢复的业务记录；会话和临时任务可硬删除。
- 关联删除默认受数据库外键保护，不做无提示级联删除。
- 内容、项目、直播和合同的重要状态变化写历史记录。
- 仪表盘指标必须能追溯到原始业务记录与时间范围。

## 7. 自动化

首版固定规则：

1. 直播前提醒负责人和跟播人员。
2. 合同到期前提醒负责人。
3. 评价被标记为负面时通知处理人。
4. 视频进入生产后 48 小时仍未定稿时预警。
5. 项目逾期、客户未回款和服务商待付款形成老板异常清单。

每次执行写入唯一幂等键，避免重启或重复调度造成重复通知。

## 8. 页面结构

- 登录与首次初始化。
- 总览：老板经营看板、个人待办和异常提醒。
- 工作：项目、任务、AI 提效。
- 内容：短视频流水线、发布日历、公众号/图书、内容数据。
- 直播电商：直播日历、直播复盘、商品、场次商品表现、客诉。
- 人员：员工、部门、考勤、绩效、重大错误。
- 商务：客户、服务商/顾问、合同、收付款。
- 分析：内容、直播、电商、人员、项目和财务看板。
- 设置：成员、角色、自动化、文件存储、组织和审计日志。

桌面端以侧栏和高密度数据表为主；移动端优先支持查看、审批和快速更新状态，不强行压缩复杂表格。

## 9. 错误与安全

- 所有写操作使用结构化校验，返回可定位字段的错误。
- 登录、初始化管理员、上传、导入与外部回调属于信任边界，必须限制速率、类型和大小。
- 密码、Token 与存储密钥只从环境变量或加密配置读取，不进入日志。
- 导入先校验再事务写入；失败时整批回滚并提供错误报告。
- 自动化失败保留错误和重试次数，不静默吞掉。

## 10. 测试与验收

- 数据模型：迁移可在空 PostgreSQL 上执行，种子数据可重复初始化。
- 权限：每个敏感域至少覆盖允许与拒绝路径。
- 状态机：合法流转成功，非法流转被拒绝并保留历史。
- 金额与指标：用固定样例验证 GMV、退款率、应收应付和薪资汇总。
- 自动化：幂等性和到期时间边界有可运行测试。
- Web：关键页面有加载、空、错误和成功状态；桌面与移动视口通过冒烟测试。
- 部署：全新环境执行 Docker Compose 后可完成初始化、登录和核心业务闭环。

## 11. 交付策略

`main` 只保存可审查的正式版本。每个模块先建 Issue，再由独立分支与 worktree 实现并创建 PR。共享架构和数据库骨架先合并，后续模块基于最新 `main` 开始，避免多个 Agent 同时修改同一 schema。
