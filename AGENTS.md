# Creator Ops 开发规则

## 产品边界

- 本项目是面向内容公司的单公司私有化部署系统，支持多成员和角色。
- 覆盖项目任务、AI 提效、短视频、直播电商、公众号与图书、行政人事、客户与服务商、经营看板和固定业务自动化。
- 不建设通用低代码平台、通用工作流设计器或多租户 SaaS。
- 示例、测试和种子数据必须虚构；禁止提交飞书原始记录、真实人员信息、合同、薪资、Token、`.env` 或导出文件。

## 工程规则

- 遵循 Ponytail full：先判断是否需要，再复用现有代码、标准库和平台能力，最后才写最小实现。
- 单体优先；未经 Issue 明确批准，不拆微服务，不引入 Redis，不增加第二套后端。
- 安全、权限、输入校验、数据丢失防护和无障碍不可为减少代码而省略。
- 业务状态由数据库枚举与约束承载，避免在页面里复制字符串判断。
- 架构或目录变化时同步更新 `README.md` 与 `docs/architecture/`。

## GitHub 协作

- 一个 Issue 对应一个 `codex/<issue-number>-short-name` 分支、一个独立 worktree、一个 Agent 和一个 PR。
- Agent 不直接合并 `main`，不改其他 Issue 的文件，不提交生成物或私有数据。
- PR 前运行与改动匹配的最小验证，并在 PR 中记录命令与结果。
